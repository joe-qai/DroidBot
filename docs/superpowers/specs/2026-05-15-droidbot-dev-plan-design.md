# DroidBot 全栈开发计划设计文档

> **版本**：v1.0
> **日期**：2026-05-15
> **状态**：已确认

***

## 1. 概述

DroidBot 是一个 AI 驱动的 Android 兼容性测试平台。当前仓库包含 10 个静态 HTML 原型页面（纯前端，无后端、无构建工具）。本设计文档规划从静态原型演进到全栈可运行平台的开发计划。

### 1.1 核心目标

1. 构建 FastAPI + Vue3 全栈平台，实现 Android 兼容性测试的完整工作流
2. AI Agent 自动生成 uiautomator2 测试脚本（成功率 >80%）
3. 多设备并发执行测试脚本，输出标准化 HTML 报告
4. 支持项目管理、脚本管理、设备管理等后台功能
5. 预留鸿蒙/iOS 系统接入能力

### 1.2 关键决策

| 决策项     | 选择                            | 理由                  |
| ------- | ----------------------------- | ------------------- |
| 开发范围    | 全栈（前端 + 后端）                   | 需要脚本执行、设备管理等后端能力    |
| 前端技术栈   | Vue3 + Vite                   | 适合后台管理类 SPA，中文社区活跃  |
| 后端技术栈   | FastAPI + SQLAlchemy + SQLite | Python 团队熟悉，轻量部署    |
| AI 生成方式 | 全自动 Agent                     | 需求明确要求 Agent 驱动脚本生成 |
| 数据库     | SQLite                        | 轻量级，单机部署，团队规模小      |
| 优先级策略   | 先跑通核心链路                       | 需求文档第6条明确要求         |
| 项目拆分    | 按功能模块纵向拆分                     | 做完一个模块就能用           |

## 2. 子项目拆分与优先级

按功能模块纵向拆分，每个子项目前后端一起做，做到"做完一个就能用"。

| Phase  | 子项目名称            | 核心交付物                                              | 验收标准                                  |
| ------ | ---------------- | -------------------------------------------------- | ------------------------------------- |
| **P0** | 项目脚手架搭建          | FastAPI 后端骨架 + Vue3+Vite 前端骨架 + SQLite 数据模型 + 基础路由 | 前后端能跑起来，API 能响应，前端能渲染侧边栏和空白页面         |
| **P1** | 项目管理 + 脚本管理      | 项目 CRUD + 脚本 CRUD + 项目-脚本关联 + 脚本代码编辑器              | 能创建项目、在项目下管理脚本、上传 .py 文件、在线编辑脚本内容     |
| **P2** | 脚本执行引擎           | uiautomator2 运行环境 + 单设备脚本执行 + 实时日志 + 状态追踪          | 能选择脚本+设备 → 执行 → 实时看到日志和状态             |
| **P3** | 设备管理             | ADB 设备发现 + 状态监控 + 设备池管理 + 分配/释放                    | 自动扫描到连接的设备、查看状态、分配给执行任务               |
| **P4** | AI 脚本生成 Agent    | LLM 配置 + uiautomator2 MCP + Agent 循环 + 脚本成功率保障     | 输入自然语言 → Agent 生成脚本 → 真机验证 → 成功率 >80% |
| **P5** | 报告管理             | 执行结果汇总 + HTML 报告生成 + 报告列表/详情                       | 执行完成后自动生成报告、查看报告详情                    |
| **P6** | Dashboard + 辅助功能 | 统计面板 + 快捷入口 + APK 管理 + LLM 配置页完善                   | Dashboard 展示实时统计，APK 能上传管理            |

### 2.1 关键约束规则

- 脚本创建时必须关联项目（P1 实现）
- 项目下有脚本时不能删除项目（P1 实现删除保护）

### 2.2 鸿蒙/iOS 预留策略

- 设备表增加 `os_type` 字段（Android/HarmonyOS/iOS），默认 Android
- 脚本表增加 `target_os` 字段，默认 Android
- 执行引擎通过 `os_type` 选择不同的执行器（当前只实现 Android 执行器）
- 接口层预留但不实现鸿蒙/iOS 执行器

## 5. AI Agent 脚本生成核心流程

### 5.1 Agent 循环设计

```
用户输入自然语言需求
        │
        ▼
┌──────────────────────────────────────────────┐
│               AI Agent 循环                    │
│                                                │
│  ① 需求解析 → LLM 理解意图，生成测试场景列表     │
│                                                │
│  ② 场景规划 → 为每个场景生成操作步骤序列          │
│                                                │
│  ③ 脚本生成 → LLM 生成 uiautomator2 代码       │
│                                                │
│  ④ 真机调试 → 在指定设备上运行脚本               │
│     - 截图采集（每步截图）                       │
│     - 日志采集（执行日志）                       │
│                                                │
│  ⑤ 结果分析 → LLM 分析截图+日志                 │
│     - 判断脚本是否正确执行                       │
│     - 识别定位失败/操作异常                      │
│                                                │
│  ⑥ 脚本修正 → 如果失败，LLM 修正脚本            │
│     - 最多重试3次                               │
│     - 每次修正后重新执行+分析                    │
│                                                │
│  ⑦ 脚本确认 → 成功率达标，保存最终脚本           │
│                                                │
└──────────────────────────────────────────────┘
        │
        ▼
保存到脚本库 → 可在脚本管理中查看/编辑/执行
```

### 5.2 uiautomator2 MCP 工具集

Agent 在生成和调试脚本时，通过 MCP 工具与真机交互：

| MCP 工具            | 功能        | 输入                       | 输出      |
| ----------------- | --------- | ------------------------ | ------- |
| `connect_device`  | 连接指定设备    | device\_id               | 连接状态    |
| `get_ui_tree`     | 获取当前界面UI树 | device\_id               | XML UI树 |
| `take_screenshot` | 截取当前界面    | device\_id               | 截图路径    |
| `execute_step`    | 执行单个操作步骤  | device\_id, action       | 执行结果    |
| `run_script`      | 运行完整脚本    | device\_id, script\_path | 执行日志    |
| `get_device_info` | 获取设备信息    | device\_id               | 设备详情    |
| `install_apk`     | 安装APK     | device\_id, apk\_path    | 安装结果    |

### 5.3 LLM 调用策略

- **需求解析**：使用较强的模型（如GLM-5.1）理解复杂需求
- **脚本生成**：使用代码能力强的模型生成 uiautomator2 代码
- **结果分析**：使用 VLM（视觉语言模型）分析截图，判断UI状态
- **配置灵活**：LLM 配置页支持切换不同模型、调整 temperature 等

### 5.4 脚本成功率保障机制

1. **重试循环**：每次失败后最多修正3次
2. **截图验证**：每步操作后截图，VLM 分析是否到达预期界面
3. **UI树校验**：操作前获取UI树，确认目标元素存在
4. **等待策略**：脚本中内置智能等待（元素出现/界面稳定）
5. **异常捕获**：所有操作包裹 try-catch，记录异常详情

## 7. 各子项目详细范围与验收标准

### 7.1 P0 - 项目脚手架搭建

**后端交付物**：

- FastAPI 应用骨架（app/main.py, app/routers/, app/services/, app/models/）
- SQLAlchemy 数据模型定义（所有表结构）
- SQLite 数据库初始化脚本
- 基础 API 路由框架（空路由占位）
- 统一错误处理中间件
- CORS 配置（允许前端开发服务器跨域）
- 配置文件管理（.env + settings.py）

**前端交付物**：

- Vue3 + Vite 项目初始化
- Pinia 状态管理 store 骨架
- Vue Router 路由配置（所有路由占位）
- Axios HTTP 客户端封装（baseURL, interceptors）
- 统一暗色主题 CSS 变量体系
- 侧边栏 + 主内容区布局组件
- 空白页面占位组件

**验收标准**：

- `uvicorn app.main:app` 后端启动成功
- `npm run dev` 前端启动成功
- 访问 `http://localhost:5173` 能看到侧边栏 + 空白主内容区
- 访问 `http://localhost:8000/docs` 能看到 OpenAPI 文档

### 7.2 P1 - 项目管理 + 脚本管理

**后端交付物**：

- 项目 CRUD API（创建/列表/详情/编辑/删除）
- 脚本 CRUD API（创建/列表/详情/编辑/删除）
- 脚本上传/下载 API（文件上传存储 + 文件下载）
- 项目-脚本关联逻辑（脚本必须关联项目）
- 项目删除保护（项目下有脚本时禁止删除）

**前端交付物**：

- 项目管理页面（列表 + 创建弹窗 + 编辑弹窗 + 删除确认）
- 脚本管理页面（项目下的脚本列表 + 上传/新建 + 编辑弹窗）
- 脚本编辑页面（Monaco Editor 代码编辑器 + 保存）
- 脚本详情页面（信息展示 + 代码预览）
- 项目删除保护提示

**验收标准**：

- 能创建/编辑/删除项目
- 能在项目下创建脚本、上传 .py 文件
- 能在线编辑脚本代码并保存
- 能下载脚本文件
- 删除有脚本的项目时提示"该项目下有脚本，无法删除"

### 7.3 P2 - 脚本执行引擎

**后端交付物**：

- 脚本执行 API（选择脚本+设备 → 执行）
- 执行状态追踪（WebSocket 实时推送日志/状态）
- 执行日志文件采集（stdout/stderr → 文件存储）
- 执行结果记录（成功/失败/超时 + 错误信息）
- uiautomator2 运行环境配置脚本

**前端交付物**：

- 新建执行页面（选择脚本 + 选择设备 → 开始执行）
- 执行监控页面（实时日志滚动 + 状态指示器 + 进度展示）

**验收标准**：

## 12. 蓝图映射与增强规划（基于 `兼容性测试自动化Agent方案.md`）

> 本节将 `兼容性测试自动化Agent方案.md`（六层Agent架构蓝图）中的高级概念映射到本方案的落地实现中，明确哪些概念可以直接融入现有阶段、哪些需要新增阶段。

### 12.1 六层架构 → 本方案映射

| 蓝图层级 | 蓝图职责 | 本方案对应模块 | 融入阶段 |
|---------|---------|--------------|---------|
| **感知层** | 屏幕截图、XML树、性能数据、设备/蓝牙/WiFi状态 | P3 设备管理（ADB扫描）+ P2 执行引擎（截图/日志采集） | P2/P3 已覆盖 |
| **理解层** | 布局解析、VLM视觉语义理解、状态机构建 | P4 AI Agent 中的 VLM 截图分析 | P4 已覆盖 |
| **决策层** | ReAct循环、场景引擎、探索策略、异常恢复 | P4 Agent 循环（需求→场景→脚本→调试→修正） | P4 已覆盖 |
| **执行层** | UI操作（uiautomator2）、环境注入、锁端模拟 | P2 脚本执行引擎 | **需增强**（见12.2） |
| **验证层** | 结构验证、规则验证、语义验证、对比验证 | P5 报告管理（简单成功/失败判断） | **需大幅增强**（见12.3） |
| **记忆层** | 基线数据、测试经验、策略库 | P1 脚本库 + P5 报告历史 | **需增强**（见12.4） |

### 12.2 执行层增强：DeviceController + 环境注入

蓝图中的 `DeviceController` 不仅是执行 uiautomator2 操作，还包含**环境注入能力**——动态调整设备环境参数来模拟不同兼容性场景：

| 环境注入能力 | 实现方式 | 融入阶段 |
|------------|---------|---------|
| 深色模式切换 | `settings put secure ui_night_mode` | **P3 增强** |
| 字体大小调整 | `settings put system font_scale` | **P3 增强** |
| 网络状态模拟 | 飞行模式 + 网络限速 | **P3 增强** |
| 蓝牙状态控制 | `service call bluetooth_manager` | **P3 增强** |
| 屏幕方向切换 | `settings put system user_rotation` | **P3 增强** |
| 性能数据采集 | CPU/内存/帧率采样 | **P2 增强** |

**增强方式**：在 P2/P3 阶段增加 `DeviceController` 服务类，封装上述环境注入方法。执行测试时，先注入环境配置 → 执行脚本 → 恢复环境。

### 12.3 验证层增强：四维验证体系

当前 P5 报告管理仅判断脚本执行的成功/失败。蓝图中的 `VerificationLayer` 提供**四维验证**：

```
综合评分 = 0.25×结构 + 0.30×语义 + 0.25×规则 + 0.20×对比
```

| 验证维度 | 检查内容 | 技术实现 | 融入阶段 |
|---------|---------|---------|---------|
| **结构验证** | 关键元素是否存在、位置是否正确、是否被遮挡 | XML树解析 + 边界检查 | **P5 增强** |
| **规则验证** | 业务规则是否符合（蓝牙状态、元素可点击、无崩溃） | 脚本断言 + 日志分析 | **P5 增强** |
| **语义验证** | VLM判断用户是否能理解界面、操作是否可发现 | VLM 截图分析 + Prompt工程 | **P4/P5 增强** |
| **对比验证** | 与基线截图对比，检测跨环境视觉差异 | VLM 双图对比 + 相似度计算 | **P5 增强** |

**数据模型增强**：`reports` 表增加四维评分字段；`executions` 表关联验证结果。

### 12.4 记忆层增强：基线管理 + 经验积累

蓝图中的 `AgentMemory` 包含基线数据、执行历史、问题记录、学习经验。本方案可增强为：

| 记忆类型 | 存储内容 | 融入阶段 |
|---------|---------|---------|
| **基线截图** | 标准环境下的界面截图 | **P5 增强**（上传基线 → 执行时对比） |
| **执行历史** | 每次执行的步骤、截图、日志 | **P2/P5 已覆盖** |
| **问题库** | 发现的问题分类、复现条件、修复建议 | **P5 增强**（报告问题明细持久化） |
| **策略优化** | 根据历史成功率调整测试策略 | **P6+ 未来迭代** |

**新增数据表**：
```
baselines          # 基线截图存储
  id, name, screenshot_path, environment_config, created_at

issues             # 问题库（从报告提取）
  id, report_id, risk_id, severity, description, location, suggestion, status
```

### 12.5 场景引擎：内置测试场景库

蓝图中的 `ScenarioEngine` 内置了参数化的测试场景。本方案可在 P4 阶段增强：

**场景模板库**（内置在数据库中）：

| 场景名称 | 类型 | 步骤 | 验证规则 | 优先级 |
|---------|------|------|---------|--------|
| 蓝牙连接测试 | 连接 | 打开蓝牙→扫描设备→配对→连接→验证 | 蓝牙状态=connected，无错误弹窗 | P0 |
| 开锁流程测试 | 功能 | 进入App→点击开锁→等待响应→验证成功 | 开锁按钮可点击，3秒内完成 | P0 |
| 离线状态测试 | 异常 | 断开网络→打开App→验证离线提示→恢复网络 | 状态显示与实际一致 | P0 |
| 深色模式适配 | 兼容 | 开启深色模式→打开App→截图→对比 | 配色对比度正常，文字可读 | P2 |
| 字体大小适配 | 兼容 | 调大字体→打开App→截图→检查截断 | 无文字截断，布局自适应 | P2 |
| 横竖屏切换 | 兼容 | 打开App→旋转屏幕→验证布局→恢复 | 布局完整可读 | P2 |

**AI 脚本生成时**：用户选择场景模板 → Agent 基于模板生成具体脚本（而非从零生成），提升成功率。
